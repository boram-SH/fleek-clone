SUMMARY:
CRITIC REVIEW OF COMBINED FLEEK RESEARCH (5 researchers), with targeted follow-up verification.

A. CONTRADICTIONS FOUND (and resolutions):
1. Watch support: reviews-community claimed the watch app is "iOS-only"; webapp-flows claimed "wearOS + Apple Watch companions". Follow-up (Clien user review surfaced via search) confirms a Galaxy Watch (wearOS) companion exists that logs sets/weight and fires rest alerts — reviews-community is wrong. Irrelevant to scope (watch is OUT anyway), but corrects the record.
2. Free-tier characterization: store-listing says "generous free tier, no aggressive paywall" while webapp-flows/reviews-community document a 4-routine cap, Pro-gated Weekly Summary / 1-Month Tendency charts, and ads introduced in 2025 even for paying subscribers. Resolution: both true at different eras — pre-acquisition generous, post-MWM monetization tightened. For a local rebuild all limits vanish; they matter only as evidence that those stats screens exist.
3. Pricing: three researchers report lifetime $69.99; the current US App Store IAP list (verified by fetch) shows Lifetime $59.99 plus multiple monthly tiers ($2.99/$3.99/$5.99) and a $15.99 yearly promo. Prices drifted; irrelevant to rebuild.
4. Developer contact: support@fleek.fit (store listing + app.fleek.fit footer) vs busking@fleek.fitness (muscle-fatigue researcher) — the latter appears stale. DLT Partners built it; MWM/Primero now distributes it (business transfer confirmed on the app's notice board). Irrelevant to rebuild.
5. Download counts (100K+ Play vs 250K+ iOS vs 352K MWM total) are different scopes, not a real conflict.
6. MOST IMPORTANT CONTRADICTION-OF-KIND: the muscle-fatigue researcher's detailed algorithm (exponential decay, tau=18–36h, K≈16.7/hard set, 7-day reset, 1.0/0.5 primary/secondary weighting) is a RECONSTRUCTION from Fitbod/Hevy conventions, explicitly not Fleek's published formula — but the dump reads as if it were Fleek's actual mechanics. Verified constraints from Fleek itself are only: fatigue is computed purely from logged sets via each exercise's target-muscle mapping, rendered on front/back anatomical figures, decays over days, has a history-scrub time slider, and Used Muscle aggregates over 7/30/90 days/total (MWM captions re-verified). Any monotonic decay model satisfying these reproduces the UX; treat the proposed algorithm as a sound implementation choice, not parity.
7. Logging-table "Previous" values: data-model assumes a Hevy-style Previous column; webapp-flows' screenshot reconstruction shows only Set | KG | Reps | Done with values pre-filled from the prior session; reviews say "previous performance visible". Resolution: implement pre-filled values (matches Fleek screenshots); a Previous column is an acceptable enhancement.

B. GAPS THAT MATTER FOR REBUILDING (follow-up results included):
1. Units: no public evidence of an lb mode; every screenshot and caption says KG (Korean-origin app). Unresolved — rebuild should store kg canonically with an lb display toggle (cheap, removes the risk).
2. Routine model depth: unknown whether a Fleek routine stores per-exercise target sets/reps/weights or just an ordered exercise list plus per-routine rest time (routine-detail screenshot shows only exercise rows + muscle figures). Recommend Hevy-style targets (RoutineSet with target weight/reps), which is a superset of both possibilities and explains the pre-filled logging screen.
3. Per-exercise Chart sub-tab: exact plotted metrics/periods unconfirmed. Since Top Records cards are 1RM MAX / VOLUME MAX / WEIGHT MAX (MWM caption re-verified: "Top Records such as 1RM MAX"), the chart almost certainly plots those series; rebuild with a line chart of best-set e1RM, max weight, and session volume with 1M/3M/1Y/All filters. Contents of Pro-gated Weekly Summary and 1-Month Tendency charts remain unknown — do a simple weekly volume/sets bar chart and don't chase parity.
4. Cardio: verified via the Blind thread — a stepmill user couldn't log because "the machine doesn't show km", and commenters log cardio elsewhere. So Fleek's cardio entry apparently expects distance; rebuild must make duration the only required field, distance/speed optional (deliberate improvement).
5. Muscle taxonomy granularity: unknown how many regions Fleek's body figure has; visible tags are coarse ("Shoulder, Triceps"). Use a ~15-group taxonomy compatible with open-source SVG body-highlighter assets and free-exercise-db's primary/secondary muscle fields.
6. Calories: shown in the summary; formula unpublished; MET-based estimate is a reconstruction. Needs user bodyweight — implies a minimal profile/bodyweight setting. One review notes no real-time calorie display during workout; treat calories as a nice-to-have estimate, clearly approximate.
7. Post-hoc editing: a review says finished Fleek records must be deleted and re-entered to fix. Rebuild should allow full editing of past workouts — improvement, not parity.
8. Rest-timer suggestion logic ("expert recommended") is unpublished; a static default (60s, configurable globally/per-routine, per-exercise override) suffices. Fleek's timer fails in background — rebuild must run it via local notification/alarm.
9. Record tab's Photos and Body sub-tabs (only webapp-flows saw them): exact body metrics unknown (likely weight/InBody numbers). Scope minimal bodyweight logging IN (needed for calories/bodyweight-exercise volume); photos OUT of MVP.
10. Volume edge rules (dumbbells count x2, some machines add bodyweight) come from a single review; implement simple weight×reps first, add per-exercise flags later.
11. Draft persistence: no researcher confirmed whether Fleek survives app kill mid-workout; rebuild must persist the in-progress session regardless.

C. SCOPE RULING: IN = exercise library w/ muscle mapping + first-class custom exercises; routines; active logging (set table, check-off, set types, memo, substitute exercise); rest timer; summary; calendar+list history with editing; per-exercise records/charts (Epley e1RM); fatigue body map + Used Muscle 7/30/90/total; settings; bodyweight log; PR detection. OUT = social/Top Percentage, trainer mode, AI/InBody/gym-equipment, celebrity routines, expert-tips content, watch/HealthKit, photos, accounts/subscriptions/ads. Optional substitutes: static strength-standard table for Top %, "fresh muscle" suggester for AI, CSV export for data portability.
